# ADR 0001: Tech-Stack & Firefly-Integration via WebSocket

**Status:** Akzeptiert  
**Datum:** 2026-06-13  
**Kontext:** Wayfinder ist die ASD-Visualisierung für den Firefly-Radar-Tracker. Wir müssen entscheiden, mit welcher Technologie und Architektur wir bauen.

---

## Problem

Wayfinder muss:
1. **Firefly-Tracks empfangen** – Live-Datenfluss von Position, Geschwindigkeit, Identität
2. **2D-Kartendarstellung** – Luftfahrzeugs auf einer Basismap anzeigen
3. **Interaktiv sein** – Zoom, Pan, Filtering für den Lotsen
4. **Cloud-native Anforderungen erfüllen** – konfigurierbar, Observability, sauberes Herunterfahren
5. **Zertifizierungsfähig sein** – klare Rückverfolgbarkeit Anforderung → Code → Test

---

## Entscheidung

**Wayfinder nutzt folgende Technologien:**

1. **Sprache:** **Go** (Golang)
   - Grund: Schnelle Iteration, gutes Ökosystem für WebSocket-Clients, Cloud-native Libraries, einfaches Deployment, gute Testabdeckungs-Tools.
   - Alternative verworfen: Rust (zu mächtig für den Client, längere Compile-Zeiten), TypeScript (Web-Framework würde die Fachlichkeit verschleieren).

2. **Frontend:** **Go + Web-Technologien** (noch zu spezifizieren in M2)
   - Der Go-Server stellt die HTML/CSS/JS-Seite bereit.
   - Die Frontend-Visualisierung nutzt wahrscheinlich Canvas/WebGL oder eine Mapping-Library (leaflet.js, Mapbox GL, etc.).
   - Diese Entscheidung wird in M2 konkretisiert (siehe ADR0002).

3. **Kommunikation:** **WebSocket** zwischen Firefly (Server) und Wayfinder (Client)
   - Firefly sendet Track-Updates als JSON oder ähnliches strukturiertes Format über WebSocket.
   - Wayfinder abonniert und rendert jeden Track-Update live.
   - Format der WebSocket-Messages wird in M1 konkretisiert (abhängig von Firefly-API).

4. **Datenmodell:** **Go-Structs** zur Abbildung von Firefly-Tracks
   - Track: `Track{ID, Position, Velocity, Identity, Timestamp, ...}`
   - Mapping zwischen Firefly-Output und Wayfinder-Darstellung.

5. **Cloud-native Patterns:**
   - Konfigurierbarkeit via Env-Vars: `FIREFLY_ADDR` (Host:Port), `MAP_CENTER_LAT/LON`, `MAP_TILE_SOURCE`, etc.
   - Health-/Readiness-Endpoint (`/health`, `/ready`).
   - Strukturierte Logs (JSON-Format oder Text, konfigurierbar).
   - Graceful Shutdown: WebSocket-Verbindung ordnungsgemäß schließen.

---

## Konsequenzen

**Positiv:**
- Go ist schlank und schnell zu entwickeln – passt zu einem Lernprojekt.
- WebSocket ist Standard für Low-Latency-Datenströme.
- Klare Architektur: Go-Backend + JavaScript-Frontend auf einer Seite.
- Cloud-native Patterns sind in Go bewährte Standards (z.B. über `net/http`, Env-Var-Libs).

**Zu beachten:**
- Frontend-Visualisierung wird in einem späteren ADR detailliert (M2).
- WebSocket-Format hängt von Firefly-API ab – M1 muss die Schnittstelle mit Firefly klären.
- Tests müssen Go-Standards folgen (`testing`, `testify` für Assertion Libraries).
- Glossar und Milestones werden bei der Konkretisierung gepflegt.

---

## Alternatives berücksichtigt

- **Rust + WASM:** Wie Firefly, aber komplexer, längere Compile-Zeiten, für einen Client überengineert.
- **TypeScript/Node.js + React:** Web-zentrisch, aber das Datenmodell würde in JavaScript landen, Typ-Safety schwächer.
- **Python:** Zu langsam für Real-Time-Rendering, nicht für Production-ASD geeignet.
