# WF2-12.7 — Serverseitige Session-Registry + Session-Limit (AP7)

> **Stufe:** S4 · **Paket:** WF2-12.7 / ADR 0009 **AP7** · Opus 4.8 (hoher Effort) ·
> **Grundlage:** ADR 0009 §5 (zustandsbehaftete Sessions), ADR 0006 (Stateless-Split,
> hier bewusst für Sessions aufgegeben), ADR 0007 (K8s/horizontale Skalierung);
> baut auf WF2-12.3 (builtin-Login), WF2-12.5 (Sliding-Session), WF2-12.6 (absolutes
> Maximum) und AP6 (Zugangs-/Mandanten-Status). **Issue #64.**

## Warum (fachlich)

Der Plattform-Betreiber hat zwei operative Bedürfnisse, die ein **rein signiertes,
stateless Cookie prinzipiell nicht erfüllen kann**:

1. **Sofort sperren.** Ein Zugang wird pausiert oder gelöscht (Vertragsende,
   kompromittierte Konsole) — die laufende Sitzung muss **sofort** enden, nicht
   erst, wenn das Cookie irgendwann abläuft. Ein signiertes Cookie lässt sich vor
   Ablauf **nicht widerrufen**.
2. **Sessions begrenzen.** „Höchstens N gleichzeitige Sitzungen je Zugang"
   (Lizenz-/Arbeitsplatz-Modell) verlangt, dass der Server aktive Sessions
   **kennt und zählt** — ein stateless Cookie ist nicht zählbar.

Beides — plus **echtes serverseitiges Logout** als Nebengewinn — erschlägt **ein**
Mechanismus: eine DB-gestützte **Session-Registry**. DB-gestützt (nicht in-memory),
weil Wayfinder auf Kubernetes/horizontale Skalierung zielt (ADR 0007): mehrere
Instanzen müssen denselben Session-Zustand sehen, sonst würden Limit und Revoke pro
Pod statt global greifen.

## Was (technisch)

### Schema (`00014_sessions.sql`)

```
sessions(
  id           TEXT PRIMARY KEY,     -- base64url(sha256(cookie-token)); nur der Hash!
  user_id      BIGINT REFERENCES users(id) ON DELETE CASCADE,
  created_at   TIMESTAMPTZ,          -- Erst-Login; verankert das absolute Maximum (WF2-12.6)
  last_seen_at TIMESTAMPTZ,          -- bei jedem Request aufgefrischt
  expires_at   TIMESTAMPTZ,          -- gleitender Idle-Ablauf
  client_meta  JSONB                 -- best-effort UA/IP (rein advisory)
)
users.session_limit INT NULL         -- NULL = Deployment-Default, 0 = unbegrenzt
```

Nur der **Hash** des Cookie-Tokens liegt in der DB — ein Datenbank-Dump liefert
keine brauchbaren Cookies (das rohe Token existiert nur im Browser). Indizes auf
`user_id` (Zählen/Revoke) und `expires_at` (Janitor).

### Cookie: signierte Session-ID statt `subject.iat.exp`

Das Cookie trägt künftig eine **signierte, punktfreie Session-ID**
(`MintSessionID`/`ParseSessionID`). Weil das Token dot-frei ist, ist es strukturell
von einem Legacy-`subject.iat.exp`-Cookie (≥ 1 Punkt im Payload) unterscheidbar —
der Authenticator verwechselt die beiden nie. Die Signatur erlaubt, ein gefälschtes
Token **ohne** DB-Zugriff zu verwerfen; die Registry autorisiert dann das echte.

### Auflösung: ein fail-closed-Query

`SessionRepo.ResolveSession` prüft in **einer** Anweisung (`UPDATE … FROM users
LEFT JOIN tenants … RETURNING subject`): Session existiert **und** unabgelaufen
**und** Zugang `active` **und** (Mandant `active` **oder** tenant-loser Admin) —
und stempelt dabei `last_seen_at`. Damit greift die **Sofort-Pause bereits über den
Status-Join** (eine pausierte Sitzung löst nicht mehr auf), selbst ohne die Zeile
zu löschen.

### Limit: atomar beim Login

`CreateSession` erzwingt das Limit in einer Transaktion gegen TOCTOU (auch über
Replicas): `pg_advisory_xact_lock(user_id)` → abgelaufene Zeilen prunen → zählen →
Policy anwenden → einfügen.

| Policy (`WAYFINDER_SESSION_LIMIT_POLICY`) | Verhalten am Limit |
|---|---|
| `reject` (Default) | N+1-ter Login → **429** (Credentials waren gültig → kein 401; kein Enumerations-Leck). Bestehende Sessions bleiben. |
| `evict_oldest` | Älteste Session wird gelöscht, neuer Login zugelassen. Die alte Konsole fliegt beim nächsten Request raus. |

Das effektive Limit ist `COALESCE(users.session_limit, WAYFINDER_SESSION_LIMIT_DEFAULT)`;
`0` = unbegrenzt (opt-in, Default aus → Verhalten wie bisher).

### Renew, Logout, Sofort-Revoke, Janitor

- **Renew** verlängert die Registry-Zeile in-place (`ExtendSession`, gekappt auf
  `created_at + MaxLifetime`, WF2-12.6); widerrufen/abgelaufen → **401**.
- **Logout** löscht die Registry-Zeile (`DeleteSession`) — echtes serverseitiges
  Logout — und räumt das Cookie.
- **Sofort-Revoke:** Pausieren eines Zugangs (`setUserStatus`) oder Admins
  (`setAdminStatus`) → `DeleteUserSessions`; Pausieren eines Mandanten
  (`setTenantStatus`) → `DeleteTenantSessions` (Kaskade). Löschen wirkt via
  `ON DELETE CASCADE`. Der Status-Join ist der **fail-closed-Backstop**, das
  explizite Löschen hält die Zählung sauber und ist der echte Revoke.
- **Janitor:** eine Hintergrund-Goroutine (`runSessionJanitor`, an den
  Shutdown-Kontext gebunden) räumt abgelaufene Zeilen; Ablauf wird zusätzlich bei
  jeder Auflösung erzwungen.

### Sanfte Übernahme (Rollout-Entscheidung)

Beim Deploy tragen offene Browser noch das alte stateless-Cookie, für das es keine
Registry-Zeile gibt. Ein solches, **noch gültig signiertes** Legacy-Cookie wird auf
seine Signatur akzeptiert (der Authenticator fällt vom Registry- auf den
Legacy-Pfad zurück) und beim **nächsten Renew** in eine Registry-Session überführt —
verankert am ursprünglichen Erst-Login (`iat`), damit das absolute Maximum weiter
zählt. Eine aktive Konsole (Renew alle 10 min) migriert also binnen Minuten.

> **Harter Schnitt jederzeit möglich:** Wer Legacy-Cookies **sofort** entwerten
> will, rotiert `WAYFINDER_SESSION_KEY` — dann scheitert jede Legacy-Signatur und
> alle melden sich einmalig neu an, direkt in eine Registry-Session.

### Env & Metriken

| Env | Default | Bedeutung |
|---|---|---|
| `WAYFINDER_SESSION_LIMIT_DEFAULT` | `0` (unbegrenzt) | Default-Limit gleichzeitiger Sessions je Zugang (opt-in). |
| `WAYFINDER_SESSION_LIMIT_POLICY` | `reject` | `reject` \| `evict_oldest`. |

Metriken: `wayfinder_active_sessions` (Gauge, Live-Count zur Scrape-Zeit),
`wayfinder_sessions_opened_total`, `wayfinder_session_logins_rejected_total`,
`wayfinder_sessions_revoked_total`, `wayfinder_sessions_expired_swept_total`.

## Bekannte Grenzen (ehrlich)

- **Ein DB-Lookup pro authentifiziertem Request** (bewusste Abkehr vom
  Stateless-Split für Sessions, ADR 0009 §5). Bei der Zielgröße (Betreiber +
  Lotsen) unkritisch; ein Session-Cache/Redis ist die dokumentierte nächste Stufe,
  **nicht** Teil von AP7.
- **Rollout-Fenster:** ein noch nicht überführtes Legacy-Cookie ist bis zur
  Konversion (nächster Renew) bzw. bis zum Ablauf (≤ TTL) **nicht widerrufbar** —
  das ist der Preis der sanften Übernahme. `WAYFINDER_SESSION_KEY`-Rotation erzwingt
  den harten Schnitt.
- `last_seen_at` wird bei **jedem** Request geschrieben (kein Throttling) — bei der
  Zielgröße vertretbar; ein `/ws`-Strom authentifiziert nur einmal beim Upgrade.
- **Proxy-Modus** (OIDC) nutzt keine lokale Registry — die Session lebt im
  vorgelagerten Proxy; AP7 betrifft nur den builtin-Modus (offener Punkt in ADR 0009,
  falls OIDC produktiv wird).
- Manueller Browser-Durchlauf (Limit greift, Pause wirft raus) gehört in den realen
  E2E-Lauf auf einem Linux-Docker-Host.
