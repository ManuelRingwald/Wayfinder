# WF2-21.2 — Scoped Fan-out: Sicht-Filter (AOI + FL-Band)

> **Stufe:** 2 (Mandanten-isolierter Datenstrom) · **Paket:** WF2-21, Häppchen
> 21.2 — **schließt WF2-21 ab** · **Einstufung:** 🔒 S4 · Opus 4.8 ·
> **Grundlage:** ADR 0005, NFR-SEC-003; baut auf WF2-21.1 (Feed-Isolation) +
> WF2-10.3b (`view_configs`/`GetEffective`).

## Warum (fachlich)

WF2-21.1 entscheidet **welche Feeds** ein Mandant sieht. 21.2 legt den
**Sicht-Ausschnitt** darüber: nur Tracks **im Interessensgebiet (AOI)** und im
**Flugflächen-Band** verlassen den Server. Per Projektentscheid ist das eine
**harte, server-seitige Grenze** (nicht nur Frontend-Declutter), aus drei Gründen:

1. **Datensparsamkeit/Compliance & Geschäft:** In einem kommerziellen
   Multi-Tenant-System ist Datenzugriff = Geld & Compliance. Wer nur seinen
   Landkreis bucht („Basic"), darf nicht den ganzen süddeutschen Luftraum in den
   Browser gestreamt bekommen.
2. **Kein F12-Leck:** Würde man alles senden und nur im Frontend ausblenden,
   könnte jeder mit den DevTools den vollen Strom abgreifen, für den er nicht
   zahlt.
3. **Last/Bandbreite:** Server-seitiges AOI/FL ist der zentrale Hebel gegen
   massive Bandbreiten-/Ressourcen-Verschwendung.

**Bewusst client-seitig bleibt der Lebenszyklus-Filter** (confirmed/tentative/
coasting): ein Einsatzleiter muss unsichere Tracks (z. B. coasting hinter einem
Berg) per Klick **einblenden** können — das dürfen wir server-seitig **nie**
wegwerfen. Echte Klassifizierung (Freund/Feind/Rettung/Polizei) wird erst ein
server-seitiges **Premium-Feature**, wenn Firefly die Tracks anreichert.

## Was (technisch)

**`pkg/broadcast`:**
- `BBox` (WGS84-Box) + **`ViewFilter{AOI *BBox, FLMinFt, FLMaxFt *float64}`**.
  `Scope` trägt jetzt zusätzlich `view *ViewFilter`; `NewScopeWithView(feedIDs,
  view)`.
- **`ViewFilter.admits(track)`** — AOI-Check exakt (Position ist Teil jedes
  Tracks); FL-Check **fail-open**: hat ein Track **keine** gemessene Flugfläche
  (`flight_level_ft` nil), wird er **zugestellt**, nie verworfen.
- **`Scope.filterView(msg)`** — ohne View-Filter unverändert (Fast-Path), sonst
  eine Kopie nur mit den zugelassenen Tracks. `broadcastTracks` wendet pro Client
  erst `AllowsFeed` (21.1, harte Isolation) und dann `filterView` (21.2) an;
  filtert die View für einen Client **alles** weg, wird für diese Batch nichts
  gesendet (der Track altert im Frontend aus / verlässt die Sicht).

**`cmd/wayfinder`:**
- `newScopeResolver(subs, views)` zieht jetzt zusätzlich
  `view_configs.GetEffective(tenantID, userID)` heran. **`resolveViewFilter`**
  mappt die `ViewConfig` → `broadcast.ViewFilter`: AOI 1:1, **FL-Band von FL in
  Fuß umgerechnet (×100)** (DB speichert Flugflächen, der Track trägt Fuß). Kein
  Config / Config ohne AOI+FL → **nil** (keine Sicht-Beschränkung, Fast-Path).
  Lookup-Fehler (≠ ErrNotFound) → Fehler ⇒ Resolver fail-closed (`403`, kein
  Stream).

**Fehlerrichtung (Projektvorgabe, kompromisslos):** „im Zweifel **senden**".
Fehlt ein Attribut für die Filter-Entscheidung (z. B. keine FL), wird der Track
ausgeliefert — ein False-Positive (etwas zu viel angezeigt) ist in
sicherheitskritischen Domänen weit besser als ein False-Negative (ein reales
Flugzeug systematisch verschluckt). Die letzte Entscheidung trifft der Mensch.

**Kein Schema-Change** (nutzt `view_configs` aus WF2-10.3b).

## Tests

- **`pkg/broadcast`** `TestViewFilterAdmits` (Tabelle): innerhalb/außerhalb AOI
  (N/E), unter/über FL-Band, **kein FL → fail-open zugelassen**, AOI-Kante
  inklusiv; `nil`-View lässt alles durch. `TestBroadcastViewScoping`: zwei Clients
  am **selben** Feed — der AOI-skopierte bekommt nur den In-AOI-Track, der
  ungefilterte beide.
- **`cmd/wayfinder`** `TestResolveViewFilter` (DB-frei): Mapping inkl. FL→Fuß
  (×100), kein Config → nil, leere Restriktion → nil, Lookup-Fehler → Fehler;
  `TestNewScopeResolverFailsClosed` zusätzlich um den View-Lookup-Fehler-Fall
  erweitert.
- **Real gegen PostgreSQL 16** `TestIntegrationResolveViewFilter`:
  `UpsertTenantDefault` mit JSONB-AOI + FL-Band → `resolveViewFilter` (über
  `GetEffective`, User-Override→Default) liefert die korrekte `ViewFilter`
  (AOI round-trip, FL in Fuß); Tenant ohne View → nil.

## Abgrenzung / Nächstes

- **Snapshot beim Connect:** der Scope (Feeds + View) wird einmalig am Handshake
  aufgelöst; Live-Re-Scoping bei Abo-/View-Änderung ist **WF2-33** (Live-Apply).
- **Kategorie server-seitig** bleibt bewusst offen bis zur Track-Anreicherung
  (Premium-Feature, koppelt an WF2-40 / Firefly-ICD).
- **Damit ist WF2-21 komplett** (21.1 Feed-Isolation + 21.2 Sicht-Filter).
- **Nächster Schritt: WF2-22 — Isolations-Testsuite** (Property-/Fuzz-Tests:
  breite, generierte „A sieht nie B"-Abdeckung über Feeds **und** AOI/FL, als
  Gate gegen Regressionen am Isolations-Prädikat).
