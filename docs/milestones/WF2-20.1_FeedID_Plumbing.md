# WF2-20.1 — `feed_id`-Durchstich (Receiver → Broadcaster → Wire)

> **Stufe:** 2 (Mandanten-isolierter Datenstrom) · **Paket:** WF2-20, Häppchen
> 20.1 · **Einstufung:** 🔒 S4 · Opus 4.8 (sicherheitskritischer Kern) ·
> **Grundlage:** ADR 0005/0007, baut auf WF2-10 (feeds-Katalog).

## Warum (fachlich)

Stufe 2 baut den **mandanten-isolierten Datenstrom**. Der erste, unverzichtbare
Baustein ist eine **Attribution**: jeder Track muss wissen, **aus welchem Feed**
er kam. Erst damit kann der scoped Fan-out (WF2-21) entscheiden, ob ein Mandant
diesen Track sehen darf (`feed ∈ subscriptions`, NFR-SEC-003). Ohne `feed_id` am
Track ist Cross-Tenant-Filterung unmöglich.

20.1 zieht diese **`feed_id`-Naht** end-to-end durch — **ohne** schon mehrere
Feeds physisch zu empfangen (das ist 20.2). Bewusst klein und risikoarm: kein
Verhaltenswechsel im Single-Tenant-Betrieb (`feed_id` bleibt dort `0` und wird
aus dem WS-JSON weggelassen).

## Was (technisch)

Datenfluss: **Receiver → Handler → Broadcaster → WS-`TrackMessage`**.

- **`pkg/receiver`** — `Config.FeedID int64` + Receiver-Feld; die Handler-Signatur
  wird zu `func(feedID int64, tracks []cat062.DecodedTrack) error`. `handleTracks`
  reicht `r.feedID` an den Handler (und loggt es). Die `feed_id` ist eine
  **Wayfinder-seitige Attribution** (welche Quelle/Gruppe), **nicht** im
  CAT062-Draht — der Decoder bleibt unberührt.
- **`pkg/broadcast`** — neuer Typ `TrackBatch{FeedID int64; Tracks []DecodedTrack}`;
  der `trackChan`/`TracksChan()` trägt jetzt einen `TrackBatch` statt nackter
  Tracks. `tracksToMessage(batch)` **stempelt `batch.FeedID` auf jede**
  `TrackMessage`. Neues Feld `TrackMessage.FeedID int64 json:"feed_id,omitempty"`
  (omitempty → Single-Tenant-Ausgabe unverändert, `feed_id=0` wird weggelassen).
- **`cmd/wayfinder/main.go`** — `Config.FeedID` aus `WAYFINDER_FEED_ID` (Default
  `0`); der Receiver bekommt `FeedID: cfg.FeedID`, der Track-Handler verpackt die
  Tracks in `broadcast.TrackBatch{FeedID: feedID, Tracks: tracks}`.

Bewusste Abgrenzung: **`feed_id` ist `int64` = `feeds.id`** des Katalogs (worauf
`subscriptions.feed_id` zeigt), damit WF2-21 direkt darauf filtern kann. Kein
Schema-Change.

## Tests

- **`pkg/receiver`** — `TestHandleTracksStampsFeedID`: ein Receiver mit `FeedID: 42`
  reicht beim Dekodieren eines minimalen CAT062-Blocks (`3E 00 06 80 19 02`) die
  `feed_id 42` an den Handler durch. Bestehende Handler-Tests auf die neue Signatur
  angepasst.
- **`pkg/broadcast`** — `TestTracksToMessage` prüft jetzt, dass `TrackBatch{FeedID:7}`
  die `7` auf die `TrackMessage` stempelt; die bestehenden Broadcast-/ADS-B-Tests
  auf `TrackBatch` umgestellt.
- Gates grün (`go build/vet/test`, `gofmt`, `scripts/pg-test.sh`); `go 1.25`
  unverändert. Reine additive Wire-Erweiterung (Frontend ignoriert `feed_id`
  vorerst; UI-Nutzung später, WF2-40).

Doku: INSTALLATION §7.1 + TECHNICAL §6.1 (`WAYFINDER_FEED_ID`, `feed_id` im
TrackMessage); Register FR-FEED-001 (Durchstich-Anteil). Milestone (diese Datei).

## Abgrenzung / Nächstes

- **20.1 empfängt weiterhin nur einen Feed.** Der eigentliche Multi-Feed-Empfang
  (N Sockets) kommt in **20.2**.
- **WF2-20.2 — Feed-Registry & Multi-Feed-Receiver:** den `feeds`-Katalog aus der
  DB lesen (`FeedRepo.List`, Felder `multicast_group`/`port`/`id`), **N Receiver**
  starten (einer je Feed, je mit seiner `feed_id`), Lifecycle/Health je Feed;
  Single-Tenant-Fallback = ein Receiver wie heute. Erfordert ein Reordering in
  `main()` (DB vor Receiver-Start).
- Danach **WF2-21** (scoped Fan-out: `broadcast()` → Prädikat
  feed∩AOI∩FL∩Kategorie) und **WF2-22** (Isolations-Negativtests „A sieht nie B").
