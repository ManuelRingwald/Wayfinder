# WF2-12.3 — Builtin-Login (`/api/login`) + Credential-Speicher

> **Stufe:** 1 (Identität & Mandanten-Grundgerüst) · **Paket:** WF2-12, Häppchen
> 12.3 — **schließt WF2-12 ab** · **Einstufung:** 🔒 S4 · Opus 4.8 ·
> **Grundlage:** ADR 0006 §5, baut auf WF2-10/11/12.1/12.2.

## Warum (fachlich)

WF2-11/12 lieferten Authentifizierung in **drei Modi** (`proxy`/`builtin`/
`none`). Der **proxy-Modus** (OIDC am Reverse-Proxy) ist seit WF2-11.2 voll
funktionsfähig — er deckt den Cloud-/Plattform-Regelfall ab. Es fehlte der
**builtin-Modus für Standalone-Deployments ohne IdP**: ein eigener
Anmelde-Endpunkt, der Nutzername/Passwort prüft und eine Session ausstellt.
Damit kann Wayfinder auch **ohne externe Identitätsinfrastruktur** mandantenfähig
betrieben werden (On-Prem, Air-Gapped, Demo).

`BuiltinAuthenticator` (WF2-11.1) konnte ein **Session-Cookie schon
validieren** — aber niemand stellte es aus. WF2-12.3 ist genau dieses fehlende
Stück: Passwort → Session.

## Was (technisch)

**Credential-Speicher (Persistenz):**
- **Migration `00003_credentials.sql`** — neue Tabelle `credentials`
  (`user_id` PK → `users(id)` `ON DELETE CASCADE`, `password_hash TEXT`,
  `updated_at`). **Bewusst getrennt von `users`:** nur lokal angemeldete Nutzer
  haben eine Zeile; OIDC-/proxy-Nutzer haben **keine** (ADR 0006 §5). Demonstriert
  zugleich die dritte Migration des In-House-Runners (Schema-Evolution).
- **`pkg/store/credentials.go`** — `CredentialRepo` mit `Set` (Upsert via
  `ON CONFLICT (user_id) DO UPDATE`) und `GetHash` (→ `ErrNotFound`, wenn der
  Nutzer kein lokales Passwort hat). Der Hash bleibt **opak**: Hashing/Verify
  liegen in `pkg/auth`, der Store persistiert nur den argon2id-PHC-String.

**Login-Handler (HTTP):**
- **`pkg/tenant/login.go`** — `LoginHandler(users, creds, cfg)`:
  - **Nur POST**; JSON `{subject, password}` (Body auf 4 KiB begrenzt via
    `http.MaxBytesReader`); leeres Subject / kaputtes JSON → `400`.
  - **Timing-gehärtet gegen Nutzer-Enumeration:** es wird **immer** ein
    `auth.VerifyPassword` ausgeführt — gegen einen einmalig berechneten
    `dummyHash`, falls Subject **oder** Credential fehlt. So dauert ein
    fehlgeschlagener Login etwa gleich lang, egal ob das Konto existiert.
  - **Erfolg** → `auth.MintSession(subject, ttl, key)` als **HttpOnly**-Cookie
    (`SameSite=Lax`, `Secure` automatisch bei TLS-Deployment), `204 No Content`.
  - **Jeder Fehlerpfad** (unbekanntes Subject, kein Credential, falsches
    Passwort) → **dasselbe `401`**, ohne zu verraten, *warum* — keine
    Account-Existenz-Leakage. Auf Fehlern wird **nie** ein Cookie gesetzt.
  - `LoginConfig` mit Defaults: Cookie-Name `wf_session`, TTL `12h`.
- **`LogoutHandler(cfg)`** — überschreibt das Cookie mit `MaxAge=-1` (sofort
  abgelaufen), `204`.

**Verdrahtung (`cmd/wayfinder/main.go`):**
- Neues Config-Feld **`SessionTTL`** (`WAYFINDER_SESSION_TTL`, Go-Duration,
  Default `12h`).
- `/api/login` + `/api/logout` werden **nur registriert, wenn Multi-Tenancy aktiv
  ist (`dbPool != nil`) und `AuthMode == builtin`** — proxy/none stellen keine
  lokalen Sessions aus. Die Routen sind **absichtlich unauthentifiziert** (sie
  geben die Session erst aus, die die Middleware danach prüft). `Secure`-Flag wird
  an TLS gekoppelt (`WAYFINDER_TLS_CERT`/`_KEY`).

## Abgrenzung / Nächstes

- **Kein** Self-Service-Registrierungs-Endpoint und **keine** Passwort-Vergabe-UI
  — Credentials werden über `CredentialRepo.Set` gesetzt; das erste Konto legt
  **WF2-13 (Admin-Bootstrap)** an. Bis dahin hat noch kein Nutzer ein Passwort,
  d. h. builtin-Login ist verdrahtet, aber erst ab WF2-13 *bedienbar* (proxy bleibt
  der voll funktionsfähige Pfad).
- **Damit ist WF2-12 (Tenant-Context) komplett:** Middleware (12.1), Server-
  Verdrahtung (12.2), builtin-Login (12.3). `pkg/auth` liefert in allen drei Modi
  ein Subject; die Middleware löst es **fail-closed** zu Tenant/Identity auf.
- **Nächster Schritt:** **WF2-13** (Admin-Bootstrap: ersten Tenant/Nutzer +
  Passwort anlegen, `/admin`-Auth-Gate), danach **Stufe 2** (mandanten-isolierter
  Stream, WF2-20/21/22 — der sicherheitskritische Kern mit Pflicht-Negativtests).
