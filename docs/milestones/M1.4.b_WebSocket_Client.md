# M1.4.b — WebSocket-Client im Frontend

## Fachlich

Die Karte ist da, aber noch leer. Der Browser muss sich mit dem Wayfinder-Backend
verbinden (über den WebSocket unter `/ws`) und die Live-Track-Daten empfangen.
Diese Verbindung ist die „Datenader" vom Backend zur ASD-Darstellung. Wichtig:
Eine stabile Verbindung, die sich selbst wiederherstellt, falls sie kurzfristig
abreißt (z.B. Mobile, Browser-Tab im Hintergrund, Netzwerk-Hiccup).

## Technisch

- **`app.js`**: Neue Funktion `connectWebSocket()` öffnet ein WebSocket auf
  `ws://` oder `wss://` (je nachdem, ob Seite über `http://` oder `https://`
  geladen wurde) zum Endpunkt `/ws`.
- **Reconnect-Logik**: Bei `onclose` oder `onerror` wartet der Client 2
  Sekunden und versucht sich dann erneut zu verbinden. Das ist einfach und
  robust.
- **Datenformat**: Eingehende Nachrichten werden als JSON geparst (erwartet:
  `{tracks: [...], time_ms: ...}` — die Struktur aus
  `pkg/broadcast.Message`) und per `console.log()` ausgegeben.
- **Error-Handling**: Ungültige JSON → `console.error()`, unerwartete
  Ausgänge → `console.warn()`.

## Verifikation

- Server starten (mit `FIREFLY_CAT062_GROUP` auf einen nicht-existierenden
  Multicast, damit keine echten CAT062-Daten reinkommen).
- Browser öffnet `/` → Karte wird angezeigt, MapLibre lädt, `app.js`
  initialisiert.
- Browser-Konsole (F12 → Console): Logs wie `"Connecting to ws://..."`,
  `"WebSocket connected"` (oder, falls kein CAT062-Feed: Verbindung wird
  geöffnet, aber es gibt keine Nachrichten — kein Fehler, nur idel).
- Wenn Backend abstürzt oder Netzwerk fällt weg: `"WebSocket disconnected,
  reconnecting in 2000 ms"` → nach 2s erneuter Connect-Versuch.
