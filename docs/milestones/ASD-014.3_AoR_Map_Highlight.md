# ASD-014.3 — AoR-Kartendarstellung + Editor (Frontend)

> **Kontext:** Dritter und abschließender Slice von **ASD-014** (flughafengebundene
> AoR-Overlay-Ebene, ADR 0021). Rein Frontend. **Kein CAT062-Bezug, keine neuen
> Env-Variablen.** Baut auf ASD-014.1 (Transform: `id`/Höhenbänder) und ASD-014.2
> (Backend: AoR-Liste am View-Config + `whoami`) auf.

## Fachlich — warum

Der Lotse soll auf einen Blick **seinen Verantwortungsbereich** (CTR/TMA) vom
umgebenden Kontext-Luftraum unterscheiden — „das kontrolliere ich" vs. „das ist
Umgebung" (ADR 0021, Ebene 2). Und der Betreiber braucht eine Oberfläche, um pro
Mandant festzulegen, welche Lufträume dazugehören.

## Technisch — wie

### Karten-Highlight (Variante A: whoami-getrieben)
- **`session.js`:** neuer Getter `aorAirspaceIds` (aus `whoami.aor_airspace_ids`,
  Default `[]`).
- **`constants.js`:** `AIRSPACE_AOR_LAYER_ID` + `AIRSPACE_AOR_COLOR` (`#00e676`,
  bewusst distinkt zu allen `AIRSPACE_GROUPS`-Farben).
- **`layers.js`:** `addAirspaceAoRLayer(map)` — eine **Linien-Ebene über der
  Airspace-Quelle** (kein separater Fetch); `aorFilter(ids)` = `['all',
  ['==',['geometry-type'],'Polygon'], ['in',['get','id'],['literal',ids]]]`.
  Leere Liste ⇒ nichts hervorgehoben.
- **`engine.js`:** Closure `aorIds`; `updateAoR(ids)` setzt den Filter (Muster von
  `applyWeatherAOI`, `mapLoaded`-Guard); Layer wird direkt nach `addAirspaceLayers`
  eingefügt (liegt über der Airspace-Linie); `updateAoR` im Handle exportiert und
  im Load-Handler + über die `aor`-Sichtbarkeitsgruppe angebunden.
- **`MapCanvas.vue`:** Reconcile `updateAoR(session.aorAirspaceIds)` nach `initMap`
  (gleicher #219-Race wie viewCenter/aoi) + Watcher auf `session.aorAirspaceIds`.
- **Legende/Toggle:** `asd.js` `layerVisibility.aor` (Default an); in
  `LayerFilterContent.vue` ein Toggle „Verantwortungsbereich (AoR)" mit
  Farb-Swatch unter den Luftraum-Gruppen.

### Admin-Editor
- **`AdminTenantDetail.vue`:** `form.aorAirspaceIds`; `v-combobox` (Chips) zum
  Eintragen/Entfernen der **stabilen OpenAIP-Luftraum-IDs**; `loadView` liest
  `aor_airspace_ids`, `buildViewDto` normalisiert (`normalizeAorIds`: Trim, Leere
  verwerfen, dedup) und sendet die Liste nur, wenn nicht leer. Gespeichert über die
  bestehende `saveTenantView` (kein neuer Endpunkt).
- **`validateView.js`:** Client-Parität zu Server-`validateView` — Anzahl ≤ 500,
  jede id ≤ 64 Zeichen, keine Steuerzeichen (`charCodeAt < 0x20`), Nicht-Array
  abgelehnt.

## Ehrliche Grenze (Editor)

Der Editor nimmt die AoR-Lufträume als **IDs** entgegen (Chips), noch **nicht** als
Namens-Picker. Ein komfortabler „Luftraum nach Namen wählen → id speichern"-Picker
bräuchte eine **mandantenübergreifende Luftraum-Liste** (der `/api/airspace`-Feed
ist request-mandanten-gescopt); das ist als Folgearbeit vorgemerkt. Die IDs sind
über `/api/airspace` (Feature-`id`, seit ASD-014.1) bzw. den OpenAIP-Datensatz
auffindbar. Der Server validiert autoritativ.

## Schnittstellen-Wirkung

**Keine** am CAT062-Draht-Vertrag. `whoami` liefert die AoR-Liste additiv
(`omitempty`); das Frontend hebt daraus hervor. `pkg/aeronautical` unangetastet.

## Ergebnis — EDDH durchgängig

Damit ist ADR 0021 end-to-end umgesetzt: Betreiber setzt EDDHs AoR (CTR/TMA) als
ID-Liste im Admin-Editor → `whoami` liefert sie → die ASD-Karte hebt genau diese
Lufträume hervor, abgesetzt vom Kontext; der Track-Scope (AOI) bleibt unabhängig.
