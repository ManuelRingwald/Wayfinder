# WF2-10.1 — Persistenz-Schicht & Migrationen (`pkg/store`)

> **Stufe:** 1 (Umsetzung) · **Paket:** WF2-10, Häppchen 10.1 · **Einstufung:**
> S3 · Sonnet 4.6 (+Opus-Review) · **Grundlage:** ADR 0006 (+ Nachtrag), ADR 0005.
> Erstes Produktivcode-Paket von Wayfinder 2.0.

## Warum (fachlich)

Wayfinder 2.0 muss viele **isolierte Mandanten** bedienen, deren Konfiguration
sich zur Laufzeit ändert. In 1.x lag die (Single-Tenant-)Konfiguration einmalig in
ENV; in 2.0 wandert dieser Zustand in eine **PostgreSQL-Datenbank**, während die
Anwendung **zustandslos** bleibt (ADR 0006 §6). `pkg/store` ist das Fundament
dafür — ohne diese Schicht gibt es weder Identität (WF2-11) noch den
mandanten-isolierten Datenstrom (WF2-21).

Dieses Häppchen legt **Verbindung + Schema + Migrationsmechanik** an; die
typsicheren Repository-Zugriffe (CRUD) folgen in WF2-10.2/10.3.

## Was (technisch)

**Neues Paket `pkg/store`:**

- `store.go`
  - `Open(ctx, dsn) (*pgxpool.Pool, error)` — pgx-Connection-Pool für Laufzeit-
    Queries; pingt beim Start (fail-fast bei Fehlkonfiguration).
  - `DSNFromEnv() (string, error)` — liest `WAYFINDER_DB_URL`; der DSN ist ein
    **Infrastruktur-Secret** und bleibt in ENV (ADR 0006 §6, 12-Factor),
    `ErrNoDSN` wenn ungesetzt.
- `migrate.go` — **minimaler, eingebetteter Migrationsrunner** (ADR 0006 Nachtrag,
  statt `goose`):
  - `//go:embed migrations/*.sql`; Marker `-- +migrate up` / `-- +migrate down`
    (forward-only, Down bleibt für manuelle Rollbacks dokumentiert).
  - `Migrate(ctx, pool)` — legt `schema_migrations` an, wendet jede noch nicht
    verzeichnete Migration in **aufsteigender Version** an, **je eine Transaktion**
    (Postgres-transaktionales DDL → kein Teilschema bei Fehler), **idempotent**.
  - `loadMigrations`/`parseMigrationName`/`upSection` — robustes Parsen:
    `NNN_name.sql`, abgelehnt werden Fehlnamen und doppelte Versionen (lautes
    Scheitern statt stillem Überspringen).
- `migrations/00001_init.sql` — das ADR-0006-Schema: `tenants`, `users`, `feeds`
  (globaler Katalog, `sensor_mix` JSONB), `subscriptions` (Tenant↔Feed; Herz der
  Cross-Tenant-Isolation, NFR-SEC-003), `view_configs` (tenant-Default +
  optionale Nutzer-Übersteuerung, Partial-Unique-Index), `entitlements`
  (Feature-Flags als Daten).

## Tests & Gates

DB-frei (in dieser Umgebung ist **kein Docker-Daemon** für Testcontainers
verfügbar; die Schema-**Anwendung** wird in WF2-10.3 in CI gegen ein echtes
Postgres getestet):

- `store_test.go`: `TestDSNFromEnv`, `TestLoadMigrations` (genau 1 Migration,
  Version 1, up legt alle 6 Tabellen an, Down ausgeschlossen),
  `TestInitMigrationDownDropsEveryTable` (Up/Down-Symmetrie),
  `TestParseMigrationName` (gültige/ungültige Namen), `TestUpSection`.

Gates grün: `go build ./...`, `go vet ./...`, `gofmt`, `go test ./...`.

## Betriebs-/Abhängigkeits-Hinweise

- **Neue Abhängigkeit:** `github.com/jackc/pgx/v5` (Pool/Treiber). **Kein**
  Migrations-Framework, **kein** `modernc.org/sqlite` (bewusst lean).
- **Go-Baseline 1.23 → 1.25** (pgx + modernes `golang.org/x/*` verlangen es;
  Dockerfile-Basis auf `golang:1.25-bookworm`, ADR 0006 Nachtrag).
- `WAYFINDER_DB_URL` wird **noch nicht** von `main` gelesen (Paket ist Library);
  Eintrag in INSTALLATION/TECHNICAL erfolgt, sobald `main` die DB verdrahtet
  (späteres Häppchen) — der laufende Binär-Betrieb ändert sich durch 10.1 nicht.
