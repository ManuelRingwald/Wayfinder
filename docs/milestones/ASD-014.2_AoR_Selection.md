# ASD-014.2 — AoR-Auswahl pro Mandant (Backend)

> **Kontext:** Zweiter Slice von **ASD-014** (flughafengebundene AoR-Overlay-Ebene,
> ADR 0021). Reiner Backend-Slice: **Datenmodell + Admin-API + whoami**. **Kein
> CAT062-Bezug, keine neuen Env-Variablen, `pkg/aeronautical` unberührt.**

## Fachlich — warum

ADR 0021 trennt den **Track-Scope** (was der Lotse *sieht*) vom
**Verantwortungsbereich** (was er *kontrolliert*, CTR/TMA). Damit das ASD den
Verantwortungsbereich hervorheben kann, muss das System pro Mandant wissen,
**welche** Lufträume dazugehören. Kernbefund aus der Datenquellen-Recherche
(ADR-0021-Nachtrag): **keine** Quelle liefert einen expliziten
Flughafen→Luftraum-Link — der muss von uns hergestellt werden. Gewählt ist
**Auswahl-Semantik Option 1**: eine explizite, pro-Mandant konfigurierte Liste
von Luftraum-Referenzen.

## Technisch — wie (Variante A: whoami-Surfacing)

Die AoR-Menge wird als Liste **stabiler OpenAIP-`id`s** (aus ASD-014.1) am
View-Config gehalten und über `whoami` an die ASD-SPA geliefert; das Frontend
hebt in Slice 3 die Features mit passender `id` hervor. **Variante A** (statt
Server-Tagging in `/api/airspace`) hält das entkoppelte `pkg/aeronautical`
unangetastet; die Liste ist klein und nicht sensibel (die Luftraum-Features gehen
ohnehin an den Browser).

- **Migration** `00021_view_configs_aor_airspace_ids.sql`: `ALTER TABLE
  view_configs ADD COLUMN aor_airspace_ids JSONB;` (nullable, NULL = keine AoR;
  Präzedenz `00015`/`00016`, JSONB wie `aoi`).
- **Store** `pkg/store/view_configs.go`: `ViewConfig.AoRAirspaceIDs []string`;
  in `viewConfigColumns`, beiden Upserts, `viewJSONParams` (Marshal →JSONB, NULL
  wenn leer) und `scanViewConfig` (JSONB→[]string).
- **Admin-API** `pkg/adminapi/adminapi.go`: `viewDTO.AoRAirspaceIDs`
  (`json:"aor_airspace_ids,omitempty"`); `validateView` prüft **Anzahl ≤ 500**,
  jede id **≤ 64 Zeichen**, keine Steuerzeichen; `normalizeAoRIDs` trimmt, verwirft
  Leere und **dedupliziert** (Reihenfolge bleibt); `toViewConfig`/`toViewDTO`
  mappen. Editierbar **ohne neue Route** über `PUT /api/admin/view` und
  `…/tenants/{id}/view`.
- **whoami** `whoamiDTO.AoRAirspaceIDs` (`omitempty`) — die effektive AoR-Liste
  wird mitgeliefert (aus `GetEffective`, respektiert Impersonation über
  `readTenant`).

## Schnittstellen-Wirkung

**Keine** am CAT062-Draht-Vertrag. Intern additiv/rückwärtskompatibel: nullable
Spalte, `omitempty`-DTO-Felder; bestehende Views ohne AoR bleiben unverändert.

## Bedienbarkeit & Ausblick

Ohne UI setzt man die Liste zunächst über die Admin-API (roher `PUT /api/admin/
view`). Der **komfortable Editor** (Lufträume nach Namen listen, anklicken →
`id` speichern) und das eigentliche **Highlight-Styling + Legende** gehören zu
**Slice 3 (Frontend)** — inkl. der Client-seitigen `validateView`-Parität für das
neue Feld. Optional dort auch Höhenband-Label/-Filter aus `lower`/`upper`
(ASD-014.1).
