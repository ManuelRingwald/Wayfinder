# WF2-11.2 — AuthN: proxy-Modus OIDC-Validierung (`pkg/auth`)

> **Stufe:** 1 (Umsetzung) · **Paket:** WF2-11, Häppchen 11.2 (Abschluss von
> WF2-11) · **Einstufung:** 🔒 S4 · Opus 4.8 · **Grundlage:** ADR 0006 §5, ADR 0003.

## Warum (fachlich)

Im primären Betriebsmodus (`proxy`, ADR 0006 §5) authentifiziert ein
vorgelagerter OIDC-Reverse-Proxy (oauth2-proxy/Ingress) den Nutzer und reicht das
OIDC-Token weiter. Wayfinder **validiert** es selbst — statt einem Header blind zu
vertrauen — und gewinnt daraus das **Subject** (Identität). Das ist
Defense-in-Depth: eine Proxy-Fehlkonfiguration, die ungeprüfte Tokens durchlässt,
wird abgefangen.

Damit kann `pkg/auth` in **allen drei Modi** (none/builtin/proxy) ein Subject
liefern — **WF2-11 (AuthN) ist abgeschlossen**. Die Abbildung Subject→Nutzer→
Mandant und die HTTP-Middleware folgen in WF2-12.

## Was (technisch)

- `proxy.go`:
  - `ProxyAuthenticator` mit einem `idTokenVerifier`-Interface (go-oidcs
    `*oidc.IDTokenVerifier` erfüllt es) — so ist die Validierung gegen einen
    lokalen Test-Issuer unit-testbar.
  - `NewProxyAuthenticator(ctx, issuer, audience)` — baut über `oidc.NewProvider`
    (Discovery + JWKS) einen Verifier; braucht beim Start Netz + Kontext, danach
    cached/refresht go-oidc die JWKS.
  - `Authenticate(r)` — liest den **Bearer-Token** (`bearerToken`,
    case-insensitives `Authorization: Bearer …`), validiert **Issuer, Audience,
    Signatur (gegen JWKS), Ablauf** via go-oidc und gibt das `sub` zurück.
    Fehlender Token / fehlgeschlagene Validierung / leeres Subject →
    `ErrUnauthenticated` (fail-closed; die genaue Ursache wird **nicht** an den
    Aufrufer geleakt).

**Dependency:** `github.com/coreos/go-oidc/v3` (mit dem Projektverantwortlichen
abgestimmt) — die kanonische OIDC-Bibliothek; zieht `go-jose/v4` und
`golang.org/x/oauth2`. **Keine** selbstgebaute JWT/JWKS-Verifikation. Go-Baseline
unverändert (1.25).

## Tests (DB-frei, ohne externes IdP)

`proxy_test.go` testet den **echten** go-oidc-Pfad gegen einen lokalen Issuer:
ein im Test erzeugter RSA-Schlüssel, ein `httptest`-JWKS-Endpoint und
selbst-signierte RS256-JWTs (Mint-Helfer ohne JWT-Lib-Abhängigkeit).

- `TestProxyAuthenticatorValid`: gültiges Token → Subject.
- `TestProxyAuthenticatorRejects`: **alle** Ablehnungsgründe →
  `ErrUnauthenticated`: fehlend, kein JWT, abgelaufen, falsche Audience, falscher
  Issuer, **falsche Signatur** (mit untrustetem Schlüssel signiert), leeres Subject.
- `TestBearerToken`: Extraktion/Negativfälle.

Gates grün: `go build/vet/test`, `gofmt`.

## Stand

**WF2-11 (AuthN) abgeschlossen** — `pkg/auth` liefert in allen drei Modi ein
Subject. **Nächstes Paket: WF2-12 (Tenant-Context-Middleware)** — Authenticator-
Factory aus `WAYFINDER_AUTH_MODE` + Config, HTTP-Middleware ruft `Authenticate`,
löst Subject→Nutzer→Mandant über `store.UserRepo.GetBySubject` auf und legt den
Mandanten **fail-closed** in den Request-Kontext; dazu der builtin-Login-Handler
(Passwort prüfen → Session-Cookie setzen).
