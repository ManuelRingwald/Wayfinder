# WF2-23.1 — Audit-Log (Scope-Authorization beim `/ws`-Connect)

> **Stufe:** 2 (Mandanten-isolierter Datenstrom) · **Paket:** WF2-23, Häppchen
> 23.1 · **Einstufung:** 🔒 S2–S3 · Sonnet 4.6 (+Opus-Review, Audit-Pfad) ·
> **Grundlage:** ADR 0005, NFR-SEC-003 („Audit-Spur"); baut auf WF2-21 (Scope).

## Warum (fachlich)

NFR-SEC-003 verlangt als letzten Baustein eine **Audit-Spur: welcher Tenant sah
welchen Scope**. In einem kommerziellen Multi-Tenant-ASD ist das zugleich
Compliance-Nachweis (wer war *autorisiert*, was zu sehen) und Grundlage fürs
Billing-/SLA-Reporting. WF2-23.1 schreibt diesen Nachweis: bei jedem `/ws`-Connect
ein **strukturiertes Audit-Event** mit Mandant, Nutzer und dem **aufgelösten
Scope** (abonnierte Feeds + AOI/FL-Band).

## Was (technisch)

**Entscheidung (Projektvorgabe):** **12-Factor mit strukturiertem `slog`** — keine
DB-Audit-Tabelle. Das ist performant beim Verbindungsaufbau, und das Querying wird
an externe Log-Senken (ELK/Datadog/…) ausgelagert.

`cmd/wayfinder/main.go`:
- `newScopeResolver(subs, views, logger)` bekommt einen Logger; intern ein
  **Audit-Sub-Logger** `logger.With("component","audit")` (für Routing/Filtern in
  der Senke).
- **`logScopeAudit`** emittiert am **Autorisierungs-Punkt** (nachdem Feeds + View
  aufgelöst sind, vor dem Upgrade) ein Event `ws scope authorized` mit:
  `event=ws_connect`, `tenant_id`, `user_id`, `subject`, `role`, `feeds` (Liste),
  `aoi` (Gruppe min/max lat/lon, falls gesetzt), `fl_min_ft`/`fl_max_ft` (falls
  gesetzt), `remote`. Damit ist jede autorisierte Verbindung mit ihrem exakten
  Scope nachvollziehbar.

**Kardinalitäts-Disziplin (Projektvorgabe):** Hochkardinale Identität (`user_id`,
`subject`, `remote`) lebt **ausschließlich** im Audit-Log — **nie** als
Metrik-Label. Das `tenant`-Label für Metriken folgt in WF2-23.2.

**Kein Schema-Change**, keine neue Abhängigkeit.

## Tests

- `cmd/wayfinder/scope_test.go` `TestScopeResolverEmitsAudit`: der Resolver wird mit
  einem JSON-`slog`-Handler in einen Buffer betrieben; das Event wird geparst und
  geprüft — `component=audit`, `event=ws_connect`, `tenant_id`/`user_id`/`subject`,
  `feeds` (2 Einträge), `aoi`-Objekt vorhanden, `fl_min_ft`/`fl_max_ft` **in Fuß**
  (10000/30000 aus FL 100/300). Bestehende Resolver-Tests auf die neue Signatur
  (Logger) angepasst.

Gates grün (`go build/vet/test`, `gofmt`, `scripts/pg-test.sh`); `go 1.25`
unverändert. Doku: TECHNICAL §6 + INSTALLATION §7 (Audit-Event/Log-Senke);
Register NFR-SEC-003 (Audit-Spur).

## Abgrenzung / Nächstes

- **Nur der Autorisierungs-Nachweis beim Connect** (das sicherheits-/
  compliance-relevante Ereignis). Ein Disconnect-Event oder durable Audit-Store
  ist bewusst nicht Teil von 23.1 (slog-only, ADR 0005/0004).
- **Nächster Schritt: WF2-23.2 — Pro-Mandant-Metriken** (`pkg/metrics` um
  Label-Unterstützung erweitern; Broadcaster zählt je Mandant verbundene Clients +
  zugestellte Tracks → `…{tenant="…"}`; `tenant_id` in den Broadcaster-`Client`).
  Schließt WF2-23 und damit **Stufe 2** ab.
