# WF2-12.4 — ASD-Login-Gate + Mandanten-Login (Browser-Rand der Karte)

> **Stufe:** 3 · **Paket:** WF2-12.4 — **Auth am ASD-Kartenpfad** ·
> **Einstufung:** S3 · Opus 4.8 · **Grundlage:** ADR 0003 (Browser-Rand/Auth),
> ADR 0014 (Multi-Tenant ist der einzige Modus, Auth *immer* an); baut auf
> WF2-12.2/12.3 (Tenant-Middleware + builtin-Login) und WF2-32 (Admin-UI).

## Warum (fachlich)

Nach ADR 0014 ist die Anmeldung **immer** aktiv. Der Admin-Pfad `/admin` hatte
seine Login-Maske; der **operative Karten-Pfad `/`** (der ASD, den der **Lotse**
benutzt) hatte **keine**:

- Ein **Mandanten-Nutzer** (Rolle `user`) konnte sich nirgends sauber per UI
  anmelden. Er landete zwar über die Admin-Login-Maske im Backend, bekam dann
  aber „Kein Zugriff auf die Administration" — die Karte selbst hatte kein
  Auth-Fenster.
- Wer `/` unauthentifiziert öffnete, sah eine **leere Karte** (der `/ws`-Upgrade
  wurde serverseitig verworfen, das Frontend versuchte still zu reconnecten) —
  **kein** Login-Hinweis, kein Weg hinein.
- Es gab **keinen Logout-Knopf** in der Oberfläche (`/api/logout` existierte,
  war aber unerreichbar).

Für einen produktiven ASD ist das untauglich: Der Lotse muss `localhost:8081`
öffnen, sich anmelden und **seine** Lage sehen — und sich abmelden können. WF2-12.4
schließt genau diese Lücke und macht den **kompletten Auth-Zyklus UI-bedienbar**
(Admin **und** Mandant): anmelden → Karte → abmelden → als anderer anmelden.

## Was (technisch)

### Backend — rollen-agnostischer Identitäts-Probe

- **`GET /api/whoami`** (`cmd/wayfinder/main.go`): hinter der **Tenant-Middleware**,
  aber **nicht** `requireAdmin`. Liefert die Identität (`subject`, `tenant_id`,
  `user_id`, `role`, `must_change_password`, `features`) für **jeden**
  authentifizierten Principal, `401` ohne gültige Sitzung. Der Admin-SPA nutzt
  weiterhin das admin-gegatete `GET /api/admin/whoami`; beide teilen sich
  Handler + DTO. Neue Methode `adminapi.Handler.WhoamiHandler()` exponiert den
  vorhandenen `whoami`-Handler zum Mounten außerhalb des Admin-Gates.
- Kein neues Auth-Modell — die Isolation bleibt vollständig serverseitig: der
  `/ws`-Strom ist ohnehin mandanten-gescopt (WF2-21), `whoami` gibt nur wieder,
  **wer** die Sitzung ist. Fail-closed: keine Identität ⇒ `401` ⇒ Login-Schirm,
  **nie** eine ungescopte Karte.

### Frontend — Auth-Gate auf der Karte

- **`AsdView.vue`** hat jetzt ein **Gate**: beim Öffnen von `/` wird die Sitzung
  über `session.probe()` (→ `/api/whoami`) aufgelöst. `status === 'loading'` →
  Spinner; `'anon'` → **Login-Maske** (`LoginCard`); `'authed'` → das Live-Lagebild.
  **`MapCanvas` und damit der `/ws`-Client mounten erst nach der Anmeldung** —
  kein unauthentifizierter WebSocket-Versuch mehr, keine leere Karte.
- **`LoginCard.vue`** (neu): wiederverwendbare, präsentationsfreie Login-Karte
  (Benutzername/Passwort, `emit('submit', …)`); der Elternteil besitzt den
  Auth-Aufruf und den Lade-/Fehlerzustand.
- **Konto-/Logout-Menü** auf der Karte (oben rechts): zeigt den angemeldeten
  Nutzer; **Abmelden** für alle, plus eine Verknüpfung zur Administration für
  Admins. **Logout-Knopf** zusätzlich im Admin-Header (`AdminView.vue`).
- **`stores/session.js`** (neu): `probe`/`login`/`logout` über
  `/api/whoami`,`/api/login`,`/api/logout`; `status ∈ {loading, authed, anon}`.
  **`stores/admin.js`** bekommt eine `logout()`-Action. Der `apiFetch`-Helfer ist
  nach **`src/api.js`** ausgelagert und von beiden Stores geteilt.

### Datenfluss (anon → authed)

```
Öffnen "/" → session.probe() → GET /api/whoami
  401 → status 'anon' → LoginCard
        submit → POST /api/login (Cookie wf_session) → re-probe
  200 → status 'authed' → MapCanvas + /ws (mandanten-gescopt)
Abmelden → POST /api/logout → status 'anon' → LoginCard
```

## Schnittstellen-Wirkung

**Keine** auf den CAT062-Draht-Vertrag (rein Browser-Rand). Additiv auf der
HTTP-API: neuer Endpunkt `GET /api/whoami` (rollen-agnostisch); `/api/login` und
`/api/logout` bestanden bereits.

## Tests

- **Go** (`pkg/adminapi/adminapi_test.go`): `TestWhoamiHandlerIsRoleAgnostic` —
  `WhoamiHandler()` liefert `200` für eine `RoleUser`-Identität und `401` ohne
  Identität (der Gate-Unterschied zum admin-gegateten Pfad).
- **Frontend** (`vitest`): `stores/__tests__/session.test.js` (probe 200/401

  `admin.test.js` (logout POSTet `/api/logout`, setzt auf unauth zurück);
  `components/__tests__/loginCard.test.js` und
  `views/__tests__/asdViewAuthGate.test.js` (Struktur-Guards via `?raw`).
- Gates: `go test/vet/gofmt` grün · `vitest` 199 grün · `vite build` grün
  (eingebettetes `internal/webui/dist` neu gebaut).

## Bekannte Grenzen / Folge-Schritte

- **Inline-Re-Login bei WS-Ablauf:** Läuft die Sitzung *während* der
  Kartenansicht ab (Default-TTL 12 h), friert die Karte bis zum Reload ein; ein
  erneutes Öffnen von `/` führt zur Login-Maske. Ein In-Place-Overlay bei einem
  `401/403`-`/ws`-Close ist ein Folge-Schritt.
- **Kundenseitiges Landing:** Der `/`-Login ist die in die Karte eingebettete
  Maske; ein dediziertes Landing/Branding ist optional.
- **Manueller Durchlauf offen:** Code + Unit-Tests + Build sind grün, der
  Browser-Klickpfad (Login → Karte → Logout → Mandanten-Login) ist noch **nicht**
  gegen einen laufenden Stack ausgeführt (kein Docker in der CI-Sandbox) — gehört
  in den realen E2E-Lauf (`docs/E2E-ABNAHME.md`).
