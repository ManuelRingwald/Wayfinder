# ASD-014.1 — OpenAIP-Transform-Anreicherung (AoR-Vorbau)

> **Kontext:** Erster Slice von **ASD-014** (flughafengebundene AoR-Overlay-Ebene,
> ADR 0021). Reiner Backend-Vorbau am OpenAIP→GeoJSON-Transform. **Kein
> CAT062-Bezug, keine neuen Env-Variablen.**

## Fachlich — warum

Für die geplante **Area-of-Responsibility-Darstellung** (der Lotse sieht *seinen*
Verantwortungsbereich hervorgehoben, ADR 0021, Ebene 2) muss der einzelne
Luftraum drei Dinge tragen, die unser Transform bisher **wegwarf**:

1. **Stabile Identität (`_id`).** Die AoR-Auswahl ist eine explizite, pro-Mandant
   konfigurierte Liste (ADR 0021, Auswahl-Semantik **Option 1**). Damit die Liste
   robust ist, referenziert sie die **stabile OpenAIP-`_id`** — nicht den freien
   Namen, der pro AIRAC-Zyklus driften kann.
2. **Höhenbänder (Floor/Ceiling).** Der Lotse braucht die vertikale Ausdehnung
   („TMA 1500 ft–FL195"); ohne sie ist ein Höhen-/FL-Filter unmöglich.
3. **ICAO-Klasse.** Für Filter/Legende (Klasse C/D) und besseres Decluttering.

## Technisch — wie

**Datei:** `pkg/aeronautical/client.go` (+ Tests in `client_test.go`).

- `openaipItem` um vier decodierte Felder erweitert: `ID` (`_id`), `ICAOClass`
  (roh, numerischer Enum), `LowerLimit`/`UpperLimit` (`*openaipLimit`).
- Neuer Typ **`openaipLimit`** = OpenAIPs Vertikalgrenze `{value, unit,
  referenceDatum}`; alle drei roh gehalten und **defensiv geparst** (`props()`;
  fehlt eine Teil-Komponente, wird sie weggelassen, nil-Empfänger → nil).
- `properties()` emittiert die neuen Felder **nur für `kind == KindAirspace`**
  und **nur wenn vorhanden**:
  - `id` (string) — die stabile `_id`
  - `icao_class` (int)
  - `lower` / `upper` als **verschachteltes Objekt** `{value, unit,
    referenceDatum}`.

**Einheiten-Enums** (roh durchgereicht, das Frontend mappt):
`unit` 0 = m, 1 = ft, 6 = FL · `referenceDatum` 0 = GND, 1 = MSL, 2 = STD.

### Bewusste Entwurfsentscheidung: Datum-Vollständigkeit (3-D-Vorbehalt)

Die Höhenbänder werden **datum-vollständig** gehalten — `value` + `unit` +
`referenceDatum` je Grenze, **nie** zu einer einzelnen skalierten Zahl
verrechnet. Grund: „FL195" oder „1500 ft GND" wird erst mit **Druck/QNH** bzw.
**Gelände/DTM** zu einer absoluten Höhe. Diese Form hält die Daten konvertierbar
für Höhen-Filter heute **und** ein mögliches **3-D-Rendering** später (MapLibre
`fill-extrusion` bzw. eine spätere Render-Ebene) — ohne dass jetzt 3-D gebaut
wird. Die eigentliche Datum-Auflösung (Terrain/QNH) ist ein späterer Schritt
(vgl. Fireflys SDPS-003 / QNH, `QNHICAO` der View-Config).

## Schnittstellen-Wirkung

**Additiv und rückwärtskompatibel.** `/api/airspace`-GeoJSON gewinnt Properties;
das Frontend ignoriert unbekannte → keine Bruchwirkung. Navaid/Waypoint-Output
**unverändert** (Felder airspace-only). **Kein** Bezug zum CAT062-Draht-Vertrag.
*Nebenwirkung:* der Inhalts-Hash der Airspace-Features ändert sich einmalig →
der nächste OpenAIP-Refresh zeigt in `ChangeSummary` (AERO-3) einmalig
„added/removed" für Lufträume (rein kosmetischer Log-Effekt).

## Tests

`pkg/aeronautical/client_test.go`:
- `TestFetchEnrichesAirspaceProperties` — Airspace mit `_id`/`icaoClass`/
  `lowerLimit`/`upperLimit` liefert `id`/`icao_class`/`lower`/`upper`; ein Airspace
  **ohne** diese Felder lässt sie weg (kein Zero-Value).
- `TestEnrichmentFieldsAreAirspaceOnly` — selbst wenn OpenAIP `_id`/`icaoClass`/
  Limits auf einem **Navaid** liefert, tauchen sie im Navaid-Output **nicht** auf
  (Rückwärtskompatibilität), und der Navaid wird weiterhin normal transformiert.
