# WF2-21.1 — Scoped Fan-out: Feed-Level-Isolation

> **Stufe:** 2 (Mandanten-isolierter Datenstrom) · **Paket:** WF2-21, Häppchen
> 21.1 · **Einstufung:** 🔒 S4–S5 · Opus 4.8 · **Grundlage:** ADR 0005
> (Isolationsgrenze), NFR-SEC-003; baut auf WF2-20 (`feed_id` pro Track) +
> WF2-12 (Tenant-Identity am `/ws`-Rand) + WF2-10.3a (`subscriptions`).

## Warum (fachlich)

Das ist **der** sicherheitskritische Kern von Wayfinder 2.0: Bisher schickte der
Broadcaster **jedem** Client **alle** Tracks (all-to-all). In einer
Multi-Mandanten-Lage ist das eine Cross-Tenant-Leckage — der Worst-Case eines
sicherheitsrelevanten Luftlagebilds (NFR-SEC-003). WF2-21.1 zieht die
**Isolationsgrenze auf Feed-Ebene** ein: ein Client erhält einen Track **nur**,
wenn sein Mandant den Feed des Tracks **abonniert** hat. Server-seitig,
fail-closed, **kein** Browser-Filtern.

## Was (technisch)

**Scope am Broadcaster** (`pkg/broadcast`):
- Neuer Typ `Scope` (Menge erlaubter `feed_id`s) + `NewScope(feedIDs)` +
  `AllowsFeed(feedID)`. **nil-Scope = unscoped** (Single-Tenant: sieht alles);
  **leerer Scope = sieht nichts** (fail-closed — ein Mandant ohne Abo bekommt
  keine Tracks).
- `Client` trägt jetzt einen `*Scope`; `RegisterClient(sendChan, scope)`.
- **Track-Zustellung getrennt von globalen Nachrichten:** der `trackChan`-Pfad
  läuft über **`broadcastTracks(batch)`** — die Message wird einmal gebaut, aber
  nur an Clients gesendet, deren `scope.AllowsFeed(batch.FeedID)` gilt. Der
  `messageChan`-Pfad (`broadcast`) bleibt **global** (CAT065-Feed-Health ist
  keine Track-Daten, kein Mandanten-Bezug).
- Da ein `TrackBatch` einem Feed entspricht (alle Tracks teilen `feed_id`), ist
  die Feed-Isolation **pro Batch pro Client** ein einzelner Mengen-Check.

**Scope-Auflösung am Handshake** (`pkg/ws`):
- `ScopeResolver func(*http.Request) (*broadcast.Scope, error)` — läuft im
  `ServeHTTP` **vor** dem WebSocket-Upgrade (damit ein Fehler noch einen
  HTTP-Status liefern kann). Fehler → **`403`, kein Stream** (fail-closed).
  nil-Resolver = Scoping aus (Single-Tenant).
- `ws.New(..., scopeResolver)`; der aufgelöste Scope geht in `RegisterClient`.

**Verdrahtung** (`cmd/wayfinder/main.go`):
- `newScopeResolver(subs feedLister)` liest die **Tenant-Identity** aus dem
  Request-Context (von der Middleware gesetzt, WF2-12) und löst die abonnierten
  Feeds via `subscriptions.ListFeedIDsByTenant(tenantID)` auf → `broadcast.NewScope`.
  Ohne Identity → Fehler (fail-closed). Nur bei aktiver Tenancy (`dbPool != nil`)
  gesetzt; sonst nil (Single-Tenant bleibt all-to-all).

**Kein Schema-Change.**

## Tests

- **Pflicht-Negativtest** (`pkg/broadcast` `TestBroadcastFeedIsolation`): zwei
  Clients mit disjunkten Scopes ({Feed 1} und {Feed 2}); ein Track auf Feed 1
  erreicht **nur** A, ein Track auf Feed 2 **nur** B — eine „Isolation-Breach"-
  Assertion schlägt fehl, falls der jeweils andere ihn doch bekäme. Das ist das
  konkrete „Mandant A bekommt nie einen Track aus B's Feed" (NFR-SEC-003).
- `TestScopeAllowsFeed`: nil=alles, Menge=nur eigene Feeds, leer=nichts.
- `cmd/wayfinder` `TestNewScopeResolver` / `…FailsClosed` (DB-frei, Fake-Lister):
  Identity + Feeds → korrekter Scope; **keine Identity → Fehler**;
  **Subscription-Lookup-Fehler → kein Scope** (fail-closed).
- Bestehende Broadcast-/WS-Tests auf die neuen Signaturen (`RegisterClient`,
  `ws.New`) angepasst; `ListFeedIDsByTenant` ist in WF2-10.3a real gegen PG
  verifiziert.

Gates grün (`go build/vet/test`, `gofmt`, `scripts/pg-test.sh`); `go 1.25`
unverändert. Doku: TECHNICAL §6 (scoped Fan-out) + INSTALLATION §7 (Abo gated
Sicht); Register NFR-SEC-003 (Stream-Enforcement Feed-Ebene + Negativtest).
